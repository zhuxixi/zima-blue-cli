# Agent 提示词模板框架设计

> 基于 coverage-pjob 执行情况的反思与改进

---

## 1. 背景与问题

### 1.1 coverage-pjob 执行中发现的问题

| 问题 | 现象 | 影响 |
|------|------|------|
| **缺乏强制验证** | Agent 基于代码分析估计覆盖率为 99%，未实际运行 pytest --cov 验证 | 数据不可信，无法确认真实进度 |
| **结束条件不明确** | Agent 在 Step 37 自主判断"任务完成"并结束 | 可能过早退出，遗漏未完成工作 |
| **缺少验收机制** | 没有明确的指标计算和验收流程 | 无法确认工作质量 |
| **规则约束力弱** | 提示为建议性质（"建议"、"可以"） | Agent 可选择性执行 |
| **工作量失控** | Agent 在一轮中完成了 120 个测试 | 可能因超时而失败，风险集中 |

### 1.2 根本原因分析

```
当前 Workflow 模板结构:
├── 任务描述（模糊）
├── 执行策略（建议性）
├── 任务列表（可选）
└── 时间管理建议（参考）

问题：缺乏强制性约束和验证机制
```

---

## 2. 新模板框架设计

### 2.1 框架结构（5 大模块）

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 提示词模板框架                       │
├─────────────────────────────────────────────────────────────┤
│ 1. 背景 (Context)                                           │
│    └─ 宏观背景信息，Agent 理解任务的大环境                    │
├─────────────────────────────────────────────────────────────┤
│ 2. 需求 (Requirements)                                      │
│    └─ 明确任务目标、范围、指标                                │
├─────────────────────────────────────────────────────────────┤
│ 3. 规则 (Rules)                                             │
│    └─ 必须遵守的硬性约束，不可违反                            │
├─────────────────────────────────────────────────────────────┤
│ 4. 验收过程 (Verification)                                  │
│    └─ 如何验证指标，必须执行的具体命令                        │
├─────────────────────────────────────────────────────────────┤
│ 5. 结束指标 (Completion Criteria)                           │
│    └─ 明确什么条件下才能结束任务                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 各模块详细设计

---

#### 模块 1: 背景 (Context)

**目的**：让 Agent 理解任务的宏观环境和上下文

**内容要点**：
- 项目概述
- 当前状态
- 历史进展（如果是续跑）
- 相关文档/报告位置

**编写原则**：
- 提供 Agent 需要的所有上下文信息
- 明确指示哪些文档必须先阅读
- 说明任务的来源和目的

**示例**：
```markdown
## 背景

你是 {{ project_name }} 项目的 {{ agent_role }}。

### 项目概况
- 项目路径: {{ project_path }}
- 当前状态: {{ current_status }}
- 目标: {{ project_goal }}

### 前期成果
{{ previous_work_summary }}

### 关键文档（必须先阅读）
- 分析报告: `{{ report_path }}`
- 任务说明: `{{ task_doc_path }}`
- **第一步：阅读以上文档后再开始工作**
```

---

#### 模块 2: 需求 (Requirements)

**目的**：明确告诉 Agent 要做什么，做到什么程度

**内容要点**：
- 具体任务描述
- 量化指标（可测量的）
- 范围边界（做什么、不做什么）
- 优先级

**编写原则**：
- 使用可量化的指标（数字、百分比）
- 明确范围边界（✅ 做、❌ 不做）
- 提供优先级排序

**示例**：
```markdown
## 需求

### 主要任务
{{ task_description }}

### 量化指标
- 指标 1: 具体数值要求
- 指标 2: 质量要求
- 指标 3: 时间/效率要求

### 范围边界
- ✅ 必须做: ...
- ✅ 必须做: ...
- ❌ 不做: ...
- ❌ 不做: ...

### 优先级
P0: ... → P1: ... → P2: ...
```

---

#### 模块 3: 规则 (Rules)

**目的**：硬性约束，Agent 必须遵守，不可违反

**关键原则**：
- 使用 **强制性语言**（必须、禁止、只能）
- 每条规则都要有 **违规后果** 说明
- 规则要 **可检查、可验证**

**规则类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| **验证规则** | 何时、如何验证 | 每完成一个模块必须立即验证 |
| **数据规则** | 数据获取方式 | 禁止估计，必须实际运行命令 |
| **提交规则** | 代码提交要求 | 每个模块验证后立即 commit |
| **工作量规则** | 每轮工作限制 | 不要在一轮中完成所有工作 |
| **工具规则** | 工具使用要求 | 必须通过工具操作文件 |

**示例**：
```markdown
## 规则（必须严格遵守）

### R1: [规则名称]
**[规则描述]**
- 违规后果: [后果说明]

### R2: [规则名称]
**[规则描述]**
- 违规后果: [后果说明]
```

**覆盖率场景的具体规则**：
```markdown
## 规则（必须严格遵守）

### R1: 增量验证规则
**每完成一个模块，必须立即验证，不能累积。**
- 违规后果: 工作无效，需要返工

### R2: 真实数据规则
**覆盖率数据必须通过实际运行 pytest --cov 获取，禁止估计。**
- 违规后果: 估计的数据不可信，必须重新验证

### R3: 提交规则
**每个模块测试通过并验证后，必须立即 git commit。**
- 违规后果: 工作进度丢失风险

### R4: 禁止过度优化规则
**不要在单轮中尝试完成所有工作，控制每轮工作量。**
- 违规后果: 可能因超时而失败，导致全部工作丢失

### R5: 必须使用工具规则
**所有文件操作必须通过工具（read_file/write_file/edit_file）完成，禁止在思考中模拟。**
- 违规后果: 代码未实际写入，工作无效
```

---

#### 模块 4: 验收过程 (Verification)

**目的**：明确如何验证指标，提供可执行的具体命令

**关键原则**：
- 提供 **具体的、可复制的命令**
- 说明 **如何判断结果**（明确的通过标准）
- 明确 **验收不通过时的处理**

**内容结构**：
```markdown
## 验收过程（必须执行）

### 验收 [N]: [验收项名称]
```bash
[具体命令]
```
**通过标准**: [明确的判断标准]

### 验收不通过处理
[处理流程]
```

**覆盖率场景的验收过程**：
```markdown
## 验收过程（必须执行）

### 验收 1: 单元测试通过性
```bash
cd {{ project_path }}
pytest tests/test_<module>_unit.py -v
```
**通过标准**: 所有测试状态为 PASSED，无 FAILED/ERROR

### 验收 2: 覆盖率验证
```bash
cd {{ project_path }}
pytest tests/test_<module>_unit.py --cov=zk.<module> --cov-report=term-missing
```
**通过标准**: 
- 覆盖率百分比 >= 70%
- 记录实际的覆盖率数字，不能使用估计值

### 验收 3: 增量覆盖率验证
```bash
cd {{ project_path }}
pytest tests/test_*_unit.py --cov=zk --cov-report=term-missing
```
**通过标准**: 整体覆盖率比当前提升至少 5%

### 验收不通过处理
如果任一项验收不通过：
1. 分析失败原因
2. 补充/修复测试
3. 重新执行验收流程
4. 禁止跳过验收直接结束
```

#### 模块 5: 结束指标 (Completion Criteria)

**目的**：明确什么条件下才能结束任务，防止过早退出

**关键原则**：
- 指标必须是 **可量化、可验证** 的
- 必须 **全部满足** 才能结束
- 明确 **结束前的最后步骤**

**内容结构**：
```markdown
## 结束指标（必须全部满足才能结束）

### 必要条件清单
- [ ] 条件 1
- [ ] 条件 2
- [ ] 条件 3
...

### 结束前必须执行的命令
```bash
[验证命令]
```

### 禁止结束的情况
- 情况 1
- 情况 2
...

### 任务完成总结格式
[输出格式要求]
```

**覆盖率场景的结束指标**：
```markdown
## 结束指标（必须全部满足才能结束）

### 必要条件清单
- [ ] 所有计划模块的测试已编写完成
- [ ] 每个模块的覆盖率已通过 `pytest --cov` 验证 >= 70%
- [ ] 所有测试通过 `pytest` 验证（无失败）
- [ ] 整体覆盖率提升 >= 5%（通过增量验收验证）
- [ ] 所有更改已 git commit
- [ ] SESSION.md 已更新，记录本次进度

### 结束前必须执行的命令
```bash
cd {{ project_path }}

# 1. 运行所有新测试
pytest tests/test_*_unit.py -v

# 2. 验证整体覆盖率
pytest tests/test_*_unit.py --cov=zk --cov-report=term-missing

# 3. 检查 git 状态
git status

# 4. 确认所有更改已提交
git log --oneline -3
```

### 禁止结束的情况
以下情况 **绝对禁止** 结束任务：
- 有测试失败未修复
- 覆盖率未实际验证（仅估计）
- 有未提交的更改
- 还有计划内的模块未完成

### 任务完成总结格式
任务真正完成后，按以下格式输出总结：
```
## ✅ 任务完成验证

### 实际完成的模块
| 模块 | 测试数 | 覆盖率 |
|------|--------|--------|
| ... | ... | ... |

### 验收验证结果
- 单元测试通过率: X/Y (100%)
- 整体覆盖率: X% → Y% (+Z%)
- Git 提交: 已提交 X 个 commit

### 下一步建议
（仅建议，不执行）
```
```

## 3. 完整示例：Coverage Workflow（重构版）

```markdown
# zk-cli 测试覆盖率提升任务

## 背景

你是 zk-cli 项目的测试覆盖率提升 Agent。

### 项目信息
- 项目路径: {{ project_path }}
- 当前整体覆盖率: {{ current_coverage }}% (来自前期分析报告)
- 目标覆盖率: {{ target_coverage }}%

### 前期分析
覆盖率分析报告已生成: `{{ project_path }}/coverage_analysis_report.md`
**第一步必须先阅读此报告**，了解当前覆盖情况和缺口分析。

---

## 需求

### 当前阶段任务
{{ current_phase }}: {{ phase_description }}

### 具体目标
为以下零覆盖/低覆盖模块编写单元测试：
{% for module in target_modules %}
- `{{ module.path }}` (当前: {{ module.current_coverage }}%, 目标: {{ module.target_coverage }}%)
{% endfor %}

### 量化指标
1. 每个模块测试覆盖率 >= 70%
2. 新增测试用例 >= {{ min_test_cases }}
3. 整体覆盖率提升 >= 5%

---

## 规则（必须遵守）

1. **增量验证**: 每完成一个模块，必须立即运行 pytest 验证，不能累积多个模块再验证
2. **真实数据**: 覆盖率必须通过 `pytest --cov` 获取，禁止估计
3. **立即提交**: 每个模块验证通过后，必须立即 `git commit`
4. **控制工作量**: 不要在一轮中完成所有工作，每轮 1-2 个模块即可
5. **工具优先**: 所有文件操作必须通过工具完成，禁止在思考中模拟代码

---

## 验收过程（必须执行）

### 模块级验收（每模块一次）
```bash
cd {{ project_path }}

# 1. 测试通过性
pytest tests/test_<module>_unit.py -v
# 通过标准: 100% PASSED

# 2. 覆盖率验证
pytest tests/test_<module>_unit.py --cov=zk.<module> --cov-report=term-missing
# 通过标准: >= 70%

# 3. 提交更改
git add .
git commit -m "feat: add unit tests for <module>"
```

### 整体验收（全部完成后）
```bash
cd {{ project_path }}

# 1. 运行所有新测试
pytest tests/test_*_unit.py -v

# 2. 验证整体覆盖率提升
pytest tests/test_*_unit.py --cov=zk --cov-report=term-missing

# 3. 生成覆盖率报告
coverage html
```

---

## 结束指标（必须全部满足）

- [ ] 所有目标模块测试已完成
- [ ] 每个模块覆盖率 >= 70%（已验证）
- [ ] 所有测试通过 pytest（无失败）
- [ ] 整体覆盖率提升 >= 5%（已验证）
- [ ] 所有更改已 git commit
- [ ] SESSION.md 已创建/更新

**只有以上全部勾选后才能结束任务！**
```

## 5. 预期效果

采用新框架后：

| 问题 | 解决方案 | 预期效果 |
|------|---------|---------|
| 缺乏强制验证 | 验收过程模块 + 具体命令 | Agent 必须执行验证命令 |
| 结束条件不明确 | 结束指标模块 + 清单 | 明确的结束条件，防止过早退出 |
| 缺少验收机制 | 验收过程模块 | 明确的指标计算和验收流程 |
| 规则约束力弱 | 规则模块 + 违规后果 | 规则有约束力，违规有后果 |
| 工作量失控 | 规则模块 + 工作量控制 | 每轮工作量可控，降低风险 |
