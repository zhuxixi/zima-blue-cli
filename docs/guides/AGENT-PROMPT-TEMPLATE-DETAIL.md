## 1. 问题深度分析

### 1.1 coverage-pjob 问题复盘

#### 问题 1: 缺乏强制验证

**现象**:
- Agent 输出: "test_global_config_unit.py: 99% 覆盖率"
- 实际: Agent 基于代码分析估计，未运行 pytest --cov
- 日志搜索: 没有找到 `pytest --cov=zk` 的实际执行记录

**根因**:
- Workflow 模板: "建议运行测试验证"
- Agent 理解: "我已经看到测试代码很完整，估计有 99% 覆盖"
- 结果: 未执行真实验证

**解决方案**:
```markdown
## 规则 R2: 真实数据规则
**覆盖率数据必须通过实际运行 pytest --cov 获取，禁止估计。**

### 验证要求
执行以下命令并记录输出：
```bash
pytest tests/test_<module>_unit.py --cov=zk.<module> --cov-report=term-missing 2>&1 | tee coverage_<module>.log
```

### 输出要求
必须在响应中粘贴命令输出的覆盖率数据，格式：
```
验证结果:
---------- coverage: platform win32, python 3.13.0 ----------
Name                    Stmts   Miss  Cover
-------------------------------------------
zk/<module>.py            149      1    99%
-------------------------------------------
TOTAL                     149      1    99%
```

### 禁止行为
- ❌ "估计覆盖率为 99%"
- ❌ "根据代码分析，覆盖率应该达到..."
- ❌ 仅粘贴测试通过结果，不粘贴覆盖率报告

### 违规后果
估计的覆盖率数据将被视为无效，必须重新执行验证。
```

#### 问题 2: 结束条件不明确

**现象**:
- Step 37 Agent 输出总结并结束
- Agent: "我已成功完成 Phase 1 的测试覆盖率提升任务"
- 实际: 未验证整体覆盖率提升，未确认是否达到目标

**根因**:
- Workflow 没有明确的"结束清单"
- Agent 自主判断"完成"标准
- 缺少强制性的完成条件检查

**解决方案**:
```markdown
## 结束指标（必须全部满足才能结束）

### 强制检查清单
在结束任务前，必须逐项检查并标记：

```markdown
## 完成自检清单

- [x] 1. 所有计划模块的测试已编写完成
  - 验证: `ls tests/test_*_unit.py`
  - 结果: 找到 4 个测试文件

- [x] 2. 每个模块覆盖率已通过 pytest --cov 验证 >= 70%
  - 验证: `pytest tests/test_*_unit.py --cov=zk --cov-report=term-missing`
  - 结果: 粘贴输出...

- [x] 3. 所有测试通过 pytest（无失败）
  - 验证: `pytest tests/test_*_unit.py -v`
  - 结果: X passed, 0 failed

- [x] 4. 整体覆盖率提升 >= 5%（已验证）
  - 之前: 34%
  - 之后: 42%
  - 提升: +8%

- [x] 5. 所有更改已 git commit
  - 验证: `git log --oneline -5`
  - 结果: 已提交 4 个 commit

- [x] 6. SESSION.md 已创建/更新
  - 验证: `cat SESSION.md`
  - 结果: 已记录进度
```

**只有以上全部勾选后才能输出总结！**

### 禁止结束的情况
如果发现以下任一情况，**立即停止结束流程**，继续工作：

| 情况 | 处理方案 |
|------|---------|
| 有测试失败未修复 | 修复失败测试，重新验收 |
| 覆盖率未实际验证 | 执行 pytest --cov，获取真实数据 |
| 有未提交的更改 | 执行 git add + git commit |
| 还有计划模块未完成 | 继续完成剩余模块 |
| 整体覆盖率提升 < 5% | 补充测试，提升覆盖率 |

### 结束流程

#### Step 1: 执行自检命令
```bash
# 运行所有检查命令，保存输出
cd {{ project_path }}
pytest tests/test_*_unit.py -v 2>&1 | tee final_test.log
pytest tests/test_*_unit.py --cov=zk --cov-report=term-missing 2>&1 | tee final_coverage.log
git status
git log --oneline -5
```

#### Step 2: 填写自检清单
根据命令输出，逐项勾选清单。

#### Step 3: 自我验证
如果清单中有未勾选项，**禁止输出总结**，继续工作。

#### Step 4: 输出总结
只有全部勾选后，才能输出总结。
```

#### 问题 3: 缺少验收机制

**现象**:
- Agent 完成代码编写后直接提交
- 没有明确的"验收通过"标准
- 不知道什么算"完成"

**解决方案**:
引入三级验收机制：

```markdown
## 验收过程（三级验收）

### 第一级: 模块级验收（每模块一次）

**触发时机**: 完成单个模块的测试编写

**验收内容**:

1. **单元测试通过性验收**
   ```bash
   cd {{ project_path }}
   pytest tests/test_<module>_unit.py -v
   ```
   **通过标准**: 
   - 所有测试状态为 PASSED
   - 无 FAILED/ERROR/SKIPPED
   - 输出格式: `X passed in Ys`

2. **模块覆盖率验收**
   ```bash
   cd {{ project_path }}
   pytest tests/test_<module>_unit.py --cov=zk.<module> --cov-report=term-missing
   ```
   **通过标准**:
   - Cover >= 70%
   - 记录实际 Cover 数值

3. **代码质量验收**
   - 检查测试代码是否有明确断言
   - 检查是否使用了合适的 mock
   - 检查测试命名是否规范

**验收通过标准**: 全部 3 项通过

**验收后动作**:
```bash
git add tests/test_<module>_unit.py
git commit -m "test: add unit tests for <module> with X% coverage"
```

---

### 第二级: 阶段级验收（每阶段一次）

**触发时机**: 完成阶段内所有模块

**验收内容**:

1. **集成测试验收**
   ```bash
   cd {{ project_path }}
   pytest tests/test_*_unit.py -v
   ```
   **通过标准**: 全部通过，无冲突

2. **阶段覆盖率验收**
   ```bash
   cd {{ project_path }}
   pytest tests/test_*_unit.py --cov=zk --cov-report=term-missing
   ```
   **通过标准**: 
   - 阶段目标模块 Cover >= 70%
   - 整体覆盖率提升 >= 5%

3. **提交记录验收**
   ```bash
   git log --oneline --since="阶段开始时间"
   ```
   **通过标准**: 
   - 每个模块有独立 commit
   - Commit message 规范

**验收通过标准**: 全部 3 项通过

**验收后动作**:
- 更新 SESSION.md
- 标记阶段完成

---

### 第三级: 项目级验收（项目结束）

**触发时机**: 所有阶段完成，准备结束任务

**验收内容**:

1. **完整测试套件验收**
   ```bash
   cd {{ project_path }}
   pytest tests/test_*_unit.py -v --tb=short
   ```
   **通过标准**: 全部通过

2. **最终覆盖率验收**
   ```bash
   cd {{ project_path }}
   pytest tests/test_*_unit.py --cov=zk --cov-report=term-missing --cov-report=html
   ```
   **通过标准**: 
   - 目标模块 Cover >= 目标值
   - 整体 Cover >= 目标值

3. **文档验收**
   - SESSION.md 完整
   - 覆盖率报告生成
   - 总结文档清晰

**验收通过标准**: 全部 3 项通过

**验收后动作**:
- 输出最终总结
- 标记任务完成
```

#### 问题 4: 规则约束力弱

**现象**:
- Workflow: "建议每完成一个模块就验证"
- Agent: 完成 4 个模块后才统一验证

**解决方案**:
添加规则强制执行机制

```markdown
## 规则（必须严格遵守）

### 规则定义格式
每条规则包含：
1. **规则 ID**: R1, R2, R3...
2. **规则描述**: 明确的行为要求
3. **验证方法**: 如何检查规则遵守情况
4. **违规后果**: 不遵守的后果
5. **强制标记**: [MUST] 表示必须遵守

### 规则列表

#### R1 [MUST]: 增量验证规则
**描述**: 每完成一个模块，必须立即验证，不能累积。

**验证方法**:
- 检查 git commit 历史，每个模块应有独立 commit
- 检查日志，每个模块后应有 pytest 输出

**违规后果**:
- 如果一次提交多个模块，要求拆分
- 如果没有模块级验证，要求补充验证

**例外情况**:
- 无例外

---

#### R2 [MUST]: 真实数据规则
**描述**: 覆盖率数据必须通过实际运行 pytest --cov 获取，禁止估计。

**验证方法**:
- 检查响应中是否有 pytest --cov 输出
- 检查覆盖率数据是否有具体数字和来源

**违规后果**:
- 估计的覆盖率视为无效
- 必须重新执行验证命令

**例外情况**:
- 无例外

---

#### R3 [MUST]: 立即提交规则
**描述**: 每个模块测试通过并验证后，必须立即 git commit。

**验证方法**:
- 检查 git log，每个模块应有 commit
- Commit 应在测试验证后立即进行

**违规后果**:
- 未提交的模块视为未完成
- 要求立即提交

**例外情况**:
- 无例外

---

#### R4 [MUST]: 工作量控制规则
**描述**: 不要在一轮中完成所有工作，控制每轮工作量。

**具体限制**:
- 每轮最多完成 2 个模块
- 如果超过，主动停止，等待下一轮

**验证方法**:
- 检查单轮完成的模块数量
- 检查 Step 数，超过 20 步应停止

**违规后果**:
- 工作量过大的轮次，要求拆分
- 可能因超时而失败

**例外情况**:
- 如果单个模块特别复杂，可以申请例外

---

#### R5 [MUST]: 工具使用规则
**描述**: 所有文件操作必须通过工具（read_file/write_file/edit_file）完成，禁止在思考中模拟。

**验证方法**:
- 检查响应中是否有 ToolCall
- 检查文件修改是否通过工具完成

**违规后果**:
- 思考中的代码视为未写入
- 要求重新使用工具写入

**例外情况**:
- 无例外
```

#### 问题 5: 工作量失控

**现象**:
- Agent 在一轮中完成了 120 个测试
- Step 37 才结束
- 风险集中，可能超时

**解决方案**:

```markdown
## 工作量控制机制

### 单轮工作量限制

| 指标 | 限制 | 说明 |
|------|------|------|
| 模块数 | <= 2 | 每轮最多完成 2 个模块 |
| Step 数 | <= 20 | 每轮最多 20 个 Step |
| 测试数 | <= 40 | 每轮新增测试不超过 40 个 |
| 时间 | <= 15 分钟 | 预计耗时不超过 15 分钟 |

### 工作量检查点

每完成一个模块，检查：
- 当前已完成模块数 >= 2?
- 当前 Step 数 >= 20?
- 如果是，立即进入结束流程

### 主动停止机制

如果达到工作量限制，主动输出：
```markdown
## 轮次工作完成

本轮完成：
- 模块: X, Y
- 测试数: XX
- Step 数: XX

已达到工作量限制，本轮结束。
剩余模块将在下一轮继续。

### 进度记录
- 已完成: A, B, C, D
- 待完成: E, F, G

### 下一轮建议
从模块 E 开始继续。
```
```

## 2. 框架核心原则

### 2.1 强制性原则

所有关键要求必须：
- 使用 `[MUST]` 标记
- 提供明确的验证方法
- 说明违规后果
- 没有例外或模糊空间

### 2.2 可验证原则

所有指标必须：
- 可通过命令验证
- 输出结果明确
- 可追溯、可审计

### 2.3 增量原则

工作必须：
- 分小批量进行
- 每批都有验证
- 及时提交保存

### 2.4 透明原则

Agent 必须：
- 展示执行的命令
- 粘贴命令输出
- 说明判断依据

## 4. 技术实现方案

### 4.1 Zima CLI 需要添加的功能

#### 功能 1: 规则强制执行标记

```yaml
# workflow 中支持规则标记
spec:
  template: |
    ## 规则
    {{ rule "R1" "增量验证规则" required=true }}
    每完成一个模块，必须立即验证。
    {{ end_rule }}
  
  rules:
    - id: R1
      name: 增量验证规则
      required: true
      validator: "git_commit_per_module"
```

#### 功能 2: 验收命令自动执行

```yaml
spec:
  verification:
    - name: 覆盖率验证
      command: "pytest tests/test_{{ module }}_unit.py --cov=zk.{{ module }}"
      expected: "Cover >= 70%"
      auto_execute: true  # Zima 自动执行并检查结果
```

#### 功能 3: 结束清单自动检查

```yaml
spec:
  completion_criteria:
    - "所有模块测试已完成"
    - "每个模块覆盖率 >= 70%"
    - "所有测试通过 pytest"
  
  # Zima 自动检查清单项
  auto_check: true
```

## 5. 待决事项讨论

### 待决事项 1: 如何确保 Agent 遵守规则？

**选项 A**: 纯提示工程（靠模板措辞）
- 优点: 简单，无需代码改动
- 缺点: 无法强制，Agent 可能忽略

**选项 B**: Zima CLI 添加规则解析和执行
- 优点: 可强制验证
- 缺点: 需要开发，增加复杂度

**选项 C**: 混合方案（提示 + 部分自动化）
- 在模板中使用强措辞
- Zima 解析关键规则并提醒
- 验收命令自动执行

**推荐**: 选项 C

---

### 待决事项 2: 多轮执行如何设计？

**选项 A**: Agent 自己控制（当前）
- Agent 判断何时结束，何时继续

**选项 B**: Zima Daemon 模式
- 15分钟循环，每轮独立

**选项 C**: Workflow 自调用
- 每轮结尾 Agent 自己启动下一轮

**选项 D**: 外部调度（Cron/Timer）
- 系统定时触发

**推荐**: 需要进一步讨论

---

### 待决事项 3: 如何防止 Agent 过早总结？

**方案**: 强制结束清单
- 在模板中明确要求清单检查
- Agent 必须逐项勾选
- 未勾选项禁止输出总结
